1. Czym są skrypty efektów?

Skrypty efektów to fragmenty kodu JavaScript przypisane do Active Effect. Działają automatycznie w określonych momentach gry (wyzwalacze) lub manualnie po kliknięciu przycisku ▶ na arkuszu efektu.

Każdy skrypt jest przechowywany jako flaga na dokumencie ActiveEffect (flags.neuroshima.scripts) i wykonywany przez klasę NeuroshimaScriptRunner. Skrypty mogą być dołączone zarówno do efektów bezpośrednio na aktorze, jak i do efektów na itemach (sztuczki, pancerze, ekwipunek).

Ważne: Skrypty z efektów na itemach są aktywne tylko gdy item jest założony/aktywny (pole system.equipped !== false). Efekty na itemach niezałożonych są pomijane.

Gdzie dodawać skrypty?

Otwórz dowolny ActiveEffect (na aktorze lub itemie) → zakładka Skrypty → Dodaj Skrypt.

Edytor skryptów oferuje:

  • Etykieta — nazwa widoczna na liście skryptów
  • Wyzwalacz — kiedy skrypt ma być uruchomiony
  • Uruchom gdy wyłączony (runIfDisabled) — checkbox; gdy zaznaczony, skrypt uruchamia się nawet jeśli efekt jest wyłączony (disabled). Przydatne np. do czyszczenia/sprzątania po wyłączeniu efektu.
  • Kod — edytor JavaScript z podświetlaniem składni (CodeMirror)

Automatyczne usuwanie itemów stworzonych przez skrypt (fromEffect)

Jeśli skrypt tworzy itemy na aktorze, możesz oflagować je tak, aby zostały automatycznie usunięte gdy efekt jest kasowany (np. po wygaśnięciu). Wystarczy ustawić flagę przy tworzeniu: 

// Trigger: immediate — stwórz item i powiąż z tym efektem
await this.actor.createEmbeddedDocuments("Item", [{
  name: "Tymczasowy bufor",
  type: "gear",
  flags: { neuroshima: { fromEffect: this.effect.id } }
}]);

Przy usunięciu efektu (deleteEffect lub wygaśnięcie) system automatycznie usuwa wszystkie itemy z flagą flags.neuroshima.fromEffect === effect.id.
Możesz też wywołać to ręcznie: await this.effect.deleteCreatedItems().
Aby pominąć auto-usuwanie przy konkretnym delete: effect.delete({ skipDeletingItems: true }).

manual Zasób "ładunki" — sprawdź i zużyj ładunek z itemu

Typowy wzorzec: sprawdź czy item ma zasób o danym kluczu, odczytaj wartość, a jeśli jest > 0 — zmniejsz o 1 i wykonaj akcję. 

// Trigger: manual — na itemie z zasobem "charges" (Ładunki, min:0, max:5)
(async () => {
  if (!this.hasItemResource(this.item, "charges")) {
    return this.notification("Brak zasobu 'charges' na tym itemie.", "error");
  }
  const current = this.getItemResource(this.item, "charges");
  if (current <= 0) {
    return this.notification("Brak ładunków!", "warn");
  }
  await this.decreaseItemResource(this.item, "charges", 1);
  await this.sendMessage(`Użyto ładunku. Pozostało: ${current - 1}.`);
})();
preWeaponShot Zasób "amunicja specjalna" — zużyj przed strzałem lub anuluj

W preWeaponShot można anulować strzał (args.cancel = true) jeśli specjalny zasób jest wyczerpany. 

// Trigger: preWeaponShot — na efekcie broni z zasobem "special_ammo"
(async () => {
  const weapon = args.weapon;  // item broni z args
  const ammo = this.getItemResource(weapon, "special_ammo");
  if (ammo === null || ammo <= 0) {
    this.notification("Brak specjalnej amunicji — strzał anulowany.", "warn");
    args.cancel = true;
    return;
  }
  await this.decreaseItemResource(weapon, "special_ammo", 1);
})();
manual Odczyt i ustawienie zasobu po ID (indeksie)

Gdy znasz stały indeks zasobu (widoczny w kolumnie # zakładki Zasoby), możesz operować po ID. Pamiętaj: po usunięciu zasobu ze środka indeksy przepakowują się. 

// Trigger: manual — odczytaj zasób #0, wyświetl info, zresetuj do min
(async () => {
  const res = this.getItemResourceById(this.item, 0);
  if (!res) {
    return this.notification("Brak zasobu pod indeksem 0.", "error");
  }
  this.notification(`${res.label}: ${res.value} / ${res.max || "∞"}`);
  // zresetuj zasób #0 do wartości minimalnej
  await this.setItemResourceById(this.item, 0, res.min);
})();
prepareData Bonus do cechy na podstawie wartości zasobu (synchroniczny)

prepareData jest synchroniczny — nie używaj await. Odczyt zasobu przez getItemResource jest synchroniczny i działa poprawnie. 

// Trigger: prepareData — bonus do Zręczności za każdy ładunek w zasobie "focus"
const focus = this.getItemResource(this.item, "focus");
if (focus && focus > 0) {
  args.dexterity = (args.dexterity || 0) + focus;
}

Argumenty args

Obiekt args jest zawsze przekazywany jako parametr funkcji skryptu. Jego zawartość zależy od wyzwalacza — szczegóły opisane są w sekcji 3.

Modyfikowanie args: Większość wyzwalaczy przekazuje args przez referencję. Modyfikując właściwości (np. args.stat = 15), zmieniasz wartości używane przez system. Nie zapomnij, że args.penalties to osobny obiekt — przypisz go bezpośrednio (args.penalties.mod -= 10).

Async vs Sync

Większość wyzwalaczy jest asynchronicznych — możesz używać await. Wyjątek:

prepareData i armorCalculation są synchroniczne. Nie używaj await, fetch, ani żadnych operacji asynchronicznych w tych triggerach — spowoduje to błędy lub zignorowanie zmian.

Rzuty kością w skryptach — porównanie metod

System udostępnia trzy warianty rzutów kością. Kluczowa różnica leży w kontekście triggera: triggery SYNC (armorCalculation) nie mogą korzystać z await, bo ich wynik musi być dostępny natychmiast. Triggery ASYNC (wszystkie pozostałe) są pełnoprawnymi funkcjami async.

4. Przykłady gotowych skryptów

Leczenie w Neuroshimie 1.5 — system nie używa tradycyjnych punktów HP. Zamiast tego każda rana to odrębny Item (type: "wound") z polem damageType (D / sD / L / sL / C / sC / K / sK). Leczenie polega na:
  • redukcji poziomu rany o N stopni (wound.reduceLevel(n))
  • usunięciu rany całkowicie (wound.heal() lub actor.healAllWounds())
  • zmniejszeniu kary rany (wound.reducePenalty(n))
W systemie Neuroshima 1.5 nie używa się bezpośredniej modyfikacji PŻ. Zamiast tego stosuj rany (applyWound) lub ich leczenie (reduceLevel, healAllWounds).
manual Propital — Wariant 1: natychmiastowo redukuje najcięższe rany (jeden skrypt)

Najprostszy wzorzec konsumabla — jeden skrypt manual bezpośrednio na itemie. Brak pośredniego efektu: zużywa sztukę i od razu redukuje każdą z najcięższych ran o 1 stopień (np. C → L, L → D, D → usunięta). 

// Trigger: manual — bezpośrednio na itemie "Propital"
(async () => {
  if (!this.item.hasQuantity || this.item.system.quantity <= 0) {
    return this.notification("Brak Propitalu!", "error");
  }
  const worst = this.actor.getWorstWounds();
  if (!worst.length) {
    return this.notification("Brak aktywnych ran.");
  }
  await this.item.reduceQuantity();
  for (const wound of worst) {
    // isMinDamage = D lub sD → leczymy (usuwamy ranę), inaczej obniżamy o 1
    if (this.isMinDamage(wound.system.damageType)) await wound.heal();
    else await wound.reduceLevel(1);   // C→L, L→D itd.
  }
  await this.sendMessage(
    `<strong>Propital</strong>: poziom najcięższych ran zredukowany.`
  );
})();
manual + startTurn Propital — Wariant 2: natychmiastowe leczenie + podtrzymane działanie przez 2 rundy

Skrypt manual (na itemie) zużywa sztukę, redukuje najcięższą ranę natychmiast, a następnie nakłada na aktora drugi efekt z czasem trwania 2 rund walki. Ten efekt posiada skrypt startTurn, który na początku każdej tury aktora redukuje o 1 stopień kolejną najcięższą ranę. Po wygaśnięciu efekt znika automatycznie.

▶ Skrypt na itemie — trigger: manual (Efekt 0 na itemie = ten skrypt; Efekt 1 na itemie = efekt podtrzymany z trigger startTurn) 

// Trigger: manual — na itemie "Propital"
// Natychmiastowo redukuje 1 ranę, potem nakłada efekt podtrzymany (2 rundy)
(async () => {
  if (!this.item.hasQuantity || this.item.system.quantity <= 0) {
    return this.notification("Brak Propitalu!", "error");
  }
  await this.item.reduceQuantity();

  // ① Natychmiastowa redukcja
  const worst = this.actor.getWorstWounds();
  for (const w of worst) {
    if (this.isMinDamage(w.system.damageType)) await w.heal();
    else await w.reduceLevel(1);
  }

  // ② Nakłada efekt podtrzymany z Efektu 1 (duration: 2 rundy)
  const sustained = this.item.effects.contents[1].convertToApplied({
    "duration.rounds":     2,
    "duration.startRound": game.combat?.round ?? 0
  });
  await this.actor.applyEffect({ effectData: [sustained] });
  await this.sendMessage(
    `<strong>Propital</strong>: zastosowany — działa przez 2 rundy.`
  );
})();

// Trigger: startTurn — uruchamia się na początku każdej tury aktora
// Redukuje najcięższą aktywną ranę o 1 stopień

manual + immediate + startTurn Propital — Wariant 3: przycisk nakłada efekt, który redukuje ranę 3 razy (1× od razu + 2× na turę)

Skrypt manual (Efekt 0 na itemie) tylko zużywa sztukę i nakłada Efekt 1 z duration.rounds: 3. Efekt 1 ma dwa skrypty: immediate odpala redukcję natychmiast po nałożeniu, a startTurn odpala ją na początku każdej z 2 kolejnych tur — łącznie 3 redukcje.

Uwaga na duration: immediate odpala się przy nałożeniu (runda startowa nie liczy się jako tura efektu). Przy duration.rounds: 2 efekt wygasa przed drugą turą, więc startTurn odpaliłby się tylko raz. Ustaw duration.rounds: 3 aby dostać 1× immediate + 2× startTurn.
Używamy this.reduceWorstWound() — helper który automatycznie pobiera najgorszą ranę, redukuje ją o 1 poziom lub usuwa jeśli to D. Nie trzeba definiować DAMAGE_ORDER ręcznie.

preApplyDamage Odporność — zmiana ciężkiej rany na lekką przed pancerzem

Trigger preApplyDamage pozwala modyfikować typ obrażeń zanim pancerz zostanie przeliczony i zanim rana zostanie stworzona. reduceDamageType przesuwa typ w dół o podaną liczbę poziomów w obrębie toru (Regular: D→L→C→K; Siniak: sD→sL→sC→sK) — minimum to D lub sD.

Uwaga: W preApplyDamage rany jeszcze nie istnieją. Modyfikujesz tylko args.damageType (string) — nie wywołuj tu applyWound ani wound.reduceLevel. 
// Wariant 1: Mutacja "Twardziel" — każdy rodzaj obrażeń obniżony o 1 poziom
// C→L, K→C, L→D, D→D (floor)  |  sC→sL, sK→sC, sL→sD, sD→sD
args.damageType = this.reduceDamageType(args.damageType, 1);
// Wariant 2: Redukcja tylko ciężkich ran (C/sC/K/sK → jeden poziom niżej)
// L i D nie są tknięte — zgodnie z zasadą "odporności na poważne rany"
if (this.isHeavyDamage(args.damageType)) {
  args.damageType = this.reduceDamageType(args.damageType, 1);
}
// Wariant 3 (przeciwny): "Klątwa" — każde trafienie wychodzi ciężej niż zwykle
// D→L, L→C, C→K, K→K (ceiling)  |  sD→sL, sL→sC, sC→sK, sK→sK

5a. postRequiredTest — Wymagany rzut dla graczy

NeuroshimaScriptRunner.postRequiredTest(params) to statyczna metoda wywoływana ze skryptów efektów (najczęściej z triggera manual). Tworzy kartę czatu z przyciskiem, który pozwala graczom otworzyć okno dialogowe rzutu dla ich postaci — bez ingerencji MG.

Typowe zastosowania: testy strachu, testy odporności na truciznę, testy wytrzymałości, testy przetrwania — wszędzie tam, gdzie to gracz (a nie MG) ma wykonać rzut w odpowiedzi na zdarzenie narracyjne.

Jak to działa — podejście WFRP-style:
1. Skrypt manual (np. na potworze) wywołuje postRequiredTest — w czacie pojawia się karta.
2. Każdy gracz klika przycisk — system otwiera dla jego postaci (identyfikowanej przez UUID) dialog rzutu.
3. Po rzucie system automatycznie aplikuje konsekwencje (onSuccess / onFailure) bezpośrednio na postać gracza — żaden efekt nie musi być zainstalowany na aktorze gracza.
4. Jeśli nie podasz onSuccess / onFailure, wynik pojawia się tylko w czacie — MG lub skrypt rollTest może zareagować ręcznie.

Metoda jest statyczna — wywołuj ją jako NeuroshimaScriptRunner.postRequiredTest(...), nie przez this.postRequiredTest(...). Jest dostępna globalnie w skryptach efektów, bo NeuroshimaScriptRunner jest wstrzyknięty do kontekstu wykonania.

Formaty onSuccess / onFailure

Parametry przyjmują kilka równoważnych form:

// 1. Krótki string — aplikuje jedną kondycję (wartość 1)
onFailure: "frightened"

// 2. Obiekt { addCondition } — to samo, bardziej jawnie
onFailure: { addCondition: "frightened" }

// 3. Obiekt z value — dla kondycji liczbowych, nakłada N stosów
onFailure: { addCondition: "bleeding", value: 3 }

// 4. Tablica — wiele kondycji jednocześnie (można mieszać)
onFailure: [
  { addCondition: "frightened" },
  { addCondition: "bleeding", value: 2 }
]
Kondycje liczbowe i value: Jeśli kondycja jest typu int (np. bleeding, stunned), pole value określa o ile stosów ją zwiększyć. Dla kondycji boolean pole value jest ignorowane — zawsze ustawia stan true. Domyślna wartość to 1.
Wbudowana walidacja kluczy kondycji: Jeśli podasz nieznany klucz kondycji, system wyświetli ostrzeżenie ui.notifications.warn i zaloguje do konsoli dostępne klucze — karta zostanie mimo to wysłana, ale konsekwencja nie zostanie zaaplikowana.

Wzorzec WFRP-style — konsekwencja bez efektu na aktorze gracza

To podstawowy i zalecany sposób używania postRequiredTest. Wystarczy jeden skrypt z triggerem manual. System automatycznie identyfikuje postać gracza przez jej UUID i aplikuje kondycje bezpośrednio po rzucie.

System używa game.user.character (postać przypisana do gracza w ustawieniach użytkownika) lub — jeśli ta nie jest ustawiona — pierwszego aktywnie wybranego tokena gracza. Postać jest identyfikowana przez UUID, więc jest odporna na reload sceny.

Zaawansowane — własna logika przez rollTest

Gdy konsekwencja jest bardziej złożona niż nałożenie kondycji (np. różna liczba ran zależna od sukcesów, leczenie innego aktora przez UUID, modyfikacja zasobów), użyj dodatkowego skryptu z triggerem rollTest. Pamiętaj, że ten skrypt działa na efektach na aktorze gracza — zatem efekt musi być zainstalowany na postaci gracza.

manual + rollTest Test Leczenia Ran — liczba sukcesów decyduje o jakości leczenia (zaawansowane)

Przykład gdy onFailure/onSuccess nie wystarczy — reagujemy na dokładną liczbę sukcesów i leczymy innego aktora (pacjenta). UUID pacjenta jest przechowywane w fladze aktora leczącego.

▶ Skrypt 1 — trigger: manual (na przedmiocie lekarza)

(async () => {
  const targets = NeuroshimaScriptRunner.getTargets();
  if (!targets.length) {
    return this.notification("Wskaż tokena pacjenta przed użyciem.", "warn");
  }

  await NeuroshimaScriptRunner.postRequiredTest({
    title:             "Test Leczenia Ran",
    description:       "Każdy sukces leczy jedną ranę pacjenta o poziom niżej.",
    testType:          "skill",
    testKey:           "woundTreatment",
    requiredSuccesses: 1,
    isOpen:            true
  });

  // Zapisz UUID pacjenta w fladze aktora gracza (lekarza)
  await this.actor.setFlag("neuroshima", `pendingTreatment_${this.effect.id}`, targets[0].uuid);
})();

▶ Skrypt 2 — trigger: rollTest (na tym samym efekcie, na aktorze gracza)

(async () => {
  if (args.rollData?.skill !== "woundTreatment") return;
  const flagKey = `pendingTreatment_${this.effect.id}`;
  const patientUuid = this.actor.getFlag("neuroshima", flagKey);
  if (!patientUuid) return;

  await this.actor.unsetFlag("neuroshima", flagKey);

  if (!args.isSuccess) {
    await this.sendMessage("Leczenie nieudane — rany bez zmian.");
    return;
  }

  const patient = await fromUuid(patientUuid);
  if (!patient) return;

  const successes = args.rollData?.successes ?? 1;
  let healed = 0;
  for (let i = 0; i < successes; i++) {
    const ok = await this.reduceWorstWound(patient);
    if (ok) healed++;
    else break;
  }

  await this.sendMessage(
    healed > 0
      ? `<strong>Leczenie udane!</strong> Wyleczono ${healed} poziom(ów) ran.`
      : "Sukces, lecz brak ran do leczenia."
  );
})();

Skrypty dialogowe (dialog trigger) — Przykłady zaawansowane

Trigger dialog wykonuje się gdy otwierany jest dialog rzutu — pozwala modyfikować pola dialogu (args.fields), ukrywać skrypt (Hide Script) lub warunkowo go aktywować (Activate Script). Skrypt dialogowy jest synchroniczny.

Logika operatorów w skryptach hide/activate:
Gdy chcesz dopasować jeden z kilku kluczy → użyj || (lub): key === "a" || key === "b"
Gdy chcesz wykluczyć kilka kluczy → użyj && (i): key !== "a" && key !== "b"
Pamiętaj: key !== "a" || key !== "b" jest zawsze true — ponieważ cokolwiek to jest, nie jest jednocześnie oboma wartościami. Używaj && przy negacji.
dialog "Ciszej niż cień" — zwiększa trudność testów wykrycia (spotting i vigilance)

Hide Script (ukryj dla wszystkich innych testów):

return (args.skill?.key !== "spotting" && args.skill?.key !== "vigilance");

⚠ Błąd: key !== "spotting" || key !== "vigilance" jest zawsze true. Poprawna negacja wielu wartości wymaga operatora &&.

Pre-Apply Script

Skrypt asynchroniczny wykonywany przed aplikacją efektu na cel (uruchamia się na kontekście źródła). Zwróć false aby zablokować aplikację. Dostępny dla Damage, Target, Aura i Area. Zmienne: actor (źródło), effect, targetActor (cel).

Kiedy używać Pre-Apply Script zamiast Filter?
Filter działa tylko dla Aura i Area — filtruje automatyczne kopiowanie do celów na scenie.
Pre-Apply Script działa dla Damage, Target, Aura i Area — uruchamia się bezpośrednio przed aplikacją i ma dostęp zarówno do źródła (actor) jak i celu (targetActor). Użyj Pre-Apply gdy decyzja zależy od stanu źródła, a Filter gdy zależy od stanu celu.
Pre-Apply Script — Blokuj aplikację gdy źródło ma 0 wytrzymałości
if (actor.system?.attributes?.stamina?.value <= 0) return false;
Pre-Apply Script — Nie nakładaj efektu gdy cel już go posiada (deduplicacja)
// Zabezpiecza przed podwójnym nałożeniem tego samego efektu
const alreadyHas = (targetActor.effects ?? []).some(e => e.name === effect.name);
return !alreadyHas;
Pre-Apply Script — Aura działa tylko gdy właściciel ma kondycję "ukryty"
// Efekt aury aplikuje się tylko gdy źródło jest ukryte (ma status hidden)
const isHidden = actor.statuses?.has("hidden") ?? false;
return isHidden;
Pre-Apply Script — Nie nakładaj efektu na siebie samego (Aura)
// Aura nie nakłada się na właściciela efektu
return actor.id !== targetActor.id;
Pre-Apply Script — Zezwól tylko gdy cel ma mniej niż połowę HP
const hp = targetActor.system?.attributes?.hp;
if (!hp) return false;
return hp.value <= hp.max / 2;

Filter

9. Zasoby itemu

Zakładka Zasoby w każdej karcie itemu pozwala definiować dowolne zasoby numeryczne (np. ładunki, wytrzymałość, amunicja specjalna). Każdy zasób ma unikalny klucz tekstowy, nazwę wyświetlaną, wartość bieżącą oraz opcjonalne granice Min i Max. Zasobami można zarządzać bezpośrednio ze skryptów efektów.

Pola zasobu

PoleTypOpis
keystringUnikalny identyfikator tekstowy (używany w helperach po kluczu). Powinien być krótki i bez spacji, np. charges, fuel.
labelstringWyświetlana nazwa zasobu (widoczna w UI i w chipach podsumowania).

minnumberDolna granica clampowania. Domyślnie 0. Ignorowane gdy zasób ma włączone Unclamped.
maxnumberGórna granica clampowania. Wartość 0 oznacza brak górnego limitu (gdy Unclamped jest wyłączone). Ignorowane gdy zasób ma włączone Unclamped.
showInSummarybooleanGdy true, zasób pojawia się jako chip w podsumowaniu itemu na karcie aktora. Przełączany kliknięciem w badge # w zakładce Zasoby.

Kontrolki UI w zakładce Zasoby

  • Badge # (po lewej) — ID zasobu (indeks tablicy, 0-based). Kliknięcie przełącza showInSummary. Gdy aktywny — badge świeci na pomarańczowo, zasób pojawia się w podsumowaniu na karcie aktora.
  • ∞ (infinity) — przycisk obok kosza. Przełącza tryb Unclamped. Gdy aktywny — Min i Max są zablokowane, silnik skryptów nie clampuje wartości. Gdy nieaktywny — wartości ustawiane przez helpery są zawsze zawierane w przedziale [min, max].
  • Kosz — usuwa zasób. Pozostałe zasoby są automatycznie przepakowywane (ID zostają przenumerowane 0,1,2,…).
Uwaga o ID (indeksach): po usunięciu zasobu ze środka tablicy, indeksy pozostałych zasobów przepakowują się — zasób który miał ID=2 może stać się ID=1. Jeśli w skryptach odwołujesz się do zasobu po ID (a nie po kluczu), sprawdź indeksy po każdej zmianie struktury zasobów. Kluczem (key) jest bezpieczniejszy i preferowany sposób odwołania.

Wyświetlanie w podsumowaniu itemu

Zasoby z włączonym showInSummary pojawiają się jako chipy w sekcji Zasoby w rozwijanym podsumowaniu itemu na karcie aktora (pod Statystykami, przed Opisem). Format chipu: Nazwa: wartość / max (max pomijany gdy 0).

manual Zużyj ładunek — sprawdź i zmniejsz zasób 
// Trigger: manual — na itemie z zasobem "charges" (Ładunki, min:0, max:5)

Helpery zacięcia broni (Jamming)

Poniższe helpery są dostępne wyłącznie w triggerach preWeaponShot, weaponJam i postWeaponShot (oraz bezpośrednio w triggerze weaponJam). Każdy z nich modyfikuje odpowiedni klucz obiektu args przekazanego przez silnik.

HelperDostępny wOpis
this.modifyJammingThreshold(args, delta) preWeaponShot Modyfikuje próg zacięcia o wartość delta. Dodatni delta = trudniej się zaciąć, ujemny = łatwiej. Wartość args.jammingThreshold nigdy nie spada poniżej 0.
this.preventWeaponJam(args) preWeaponShot Ustawia args.forceNoJam = true. Broń nie może się zaciąć w tej turze niezależnie od wyniku kości. Wyższy priorytet niż forceJam.
this.forceWeaponJam(args) preWeaponShot Ustawia args.forceJam = true. Broń zawsze się zacina, nawet jeśli kości dały wynik poniżej progu zacięcia. Ignorowane gdy forceNoJam jest aktywne.
this.allowShotDespiteJam(args, count=1) weaponJam Ustawia args.canFireDespiteJam = true. Aktor oddaje strzał mimo zacięcia — broń nadal zostaje zacięta po strzale.
count: maksymalna liczba pocisków (domyślnie 1).
Strzał trafia jeśli rzut byłby sukcesem (args.wouldSucceed = true). Najczęściej sprawdzaj przez this.isStandardJam(args, min, max), które automatycznie filtruje po sukcesie.

this.clearWeaponJam(args) weaponJam Ustawia args.clearJam = true. Zacięcie jest całkowicie anulowane — broń strzela normalnie i nie zostaje oznaczona jako zacięta. Automatycznie czyści też istniejącą flagę system.jammed jeśli broń była wcześniej zacięta.

Przykład: isStandardJam(args, 11, 18) — standardowe i był sukces, isStandardJam(args, 19, 20) — krytyczne i był sukces.
Kolejność wykonania helperów:
preWeaponShot → sprawdzenie progu → (jeśli zacięcie) weaponJam → oddanie/pominięcie strzału → postWeaponShot.
Helper preventWeaponJam wywołany w preWeaponShot całkowicie zapobiega uruchomieniu weaponJam.

11. Modyfikacje broni i pancerza (weapon-mod / armor-mod)

Itemom typu weapon-mod i armor-mod można przypisywać Active Effects ze skryptami dokładnie tak samo jak każdemu innemu itemowi. Jednak ze względu na specyficzny model dołączania modyfikacji, sposób wykonania tych efektów różni się od standardowych efektów na broniach czy pancerzach.

Model propagacji efektów

Gdy modyfikacja jest dołączona (attached) do broni lub pancerza, system automatycznie kopiuje wszystkie efekty z modyfikacji na broń/pancerz-hosta. Każda taka kopia jest oznaczona flagą flags.neuroshima.fromModId = <id_modyfikacji>.

Zasada: Oryginalne efekty na samym itemie weapon-mod / armor-mod nigdy nie są wykonywane bezpośrednio. Wykonywane są tylko skopiowane kopie na hoście, i tylko wtedy gdy modyfikacja jest aktualnie dołączona (attached === true).

Kontekst this.item w skryptach modyfikacji

W skryptach efektów z modyfikacji this.item wskazuje na broń lub pancerz-host, a nie na samą modyfikację. Dzięki temu skrypty mogą bezpośrednio odczytywać i modyfikować dane broni/pancerza, do której modyfikacja jest przymocowana.

this.actorActorAktor posiadający broń/pancerz z dołączoną modyfikacją
this.itemItemBroń lub pancerz-host (NIE sama modyfikacja)
this.effectActiveEffectPropagowana kopia efektu z modyfikacji (z flagą fromModId), znajdująca się na hoście

Aby uzyskać dostęp do danych samej modyfikacji z poziomu skryptu, możesz odczytać snapshot ze słownika mods na hoście lub pobrać item modyfikacji przez fromUuid: 

// this.item = broń/pancerz-host
// this.effect.getFlag("neuroshima", "fromModId") = id modyfikacji
const modId = this.effect.getFlag("neuroshima", "fromModId");
const modSnap = this.item.system?.mods?.[modId]; // snapshot modyfikacji

// Odczyt danych broni-hosta
const weaponDmg = this.item.system.damage;       // obrażenia bazowe broni
const weaponAtk = this.item.system.attack;        // atak bazowy broni

Zasoby modyfikacji — @mod.resources vs @item.resources

Jeśli modyfikacja dodaje własne zasoby do hosta (zakładka Zasoby na karcie modyfikacji), są one propagowane na hosta z dodatkowym polem _fromModId wskazującym źródłową modyfikację.

Problem kolizji kluczy

Gdy dwie lub więcej modyfikacji doda zasoby z tym samym kluczem (np. obie mają zasób o kluczu charges), użycie @item.resources.charges.value zwróci wartość z pierwszej pasującej pozycji — niezależnie od tego, która modyfikacja jest aktualnie aktywna.

System ostrzeże Cię przy montowaniu — jeśli modyfikacja tworzy kolizję kluczy zasobów z istniejącymi zasobami na hoście, pojawi się powiadomienie UI z listą kolidujących kluczy oraz sugestią użycia @mod.resources.key.value.

Rozwiązanie: @mod.resources.key.prop

Placeholder @mod.resources.<key>.<prop> rozwiązuje zasób z hosta filtrując jednocześnie po _fromModId — każda modyfikacja dostaje swój własny zasób nawet gdy klucze się pokrywają.

PlaceholderZachowanie
@item.resources.charges.value Zwraca wartość pierwszego zasobu o kluczu charges na hoście — może trafić w zasób innej modyfikacji.
@mod.resources.charges.value Zwraca wartość zasobu o kluczu charges tej konkretnej modyfikacji (_fromModId musi pasować). Bezpieczne nawet gdy wiele modów ma ten sam klucz.

Helpery mod-scoped — bezpieczny dostęp bez ręcznego filtrowania

System dostarcza dedykowane helpery które automatycznie scopują do własnej modyfikacji efektu (via fromModId na efekcie). Nie ma potrzeby ręcznego resources.find().

Cykl życia efektów modyfikacji

  • Dołączenie modyfikacji — efekty modyfikacji są kopiowane na hosta. Wyzwalacze applyEffect i immediate uruchamiają się w tym momencie z kontekstem hosta.
  • Odłączenie modyfikacji — kopie efektów są usuwane z hosta. Wyzwalacz deleteEffect uruchamia się w tym momencie.
  • Zmiana danych — po każdej zmianie wartości w modyfikacji, kopie efektów na hoście są usuwane i tworzone od nowa (z aktualnymi danymi skryptu z oryginału).

Wyświetlanie w arkuszu aktora

Efekty z dołączonych modyfikacji są wyświetlane w liście efektów aktora jako efekty należące do broni lub pancerza-hosta (z ikoną i nazwą hosta). Efekty na samym itemie weapon-mod/armor-mod są ukryte w arkuszu aktora — nigdy nie pojawiają się na liście jako efekty samej modyfikacji. Gdy modyfikacja jest odłączona (attached === false), jej efekty w ogóle nie figurują na liście aktora.

Wyzwalacze — wszystkie obsługiwane

Wszystkie standardowe wyzwalacze działają poprawnie dla efektów modyfikacji. Poniżej kilka uwag szczególnych:

  • prepareData, armorCalculation, dialog, preRollTest, rollTest itd. — uruchamiane przez getScripts(); wymagają żeby modyfikacja była attached === true i żeby host był założony (respektują equipTransfer).
  • applyEffect / immediate — uruchamiane przy _onCreate propagowanej kopii (tj. przy dołączaniu modyfikacji). this.item = host, this.actor = właściciel hosta.
  • deleteEffect — uruchamiany przy _onDelete propagowanej kopii (tj. przy odłączaniu). Analogiczny kontekst.
  • manual — dostępny przez przycisk ▶ na efekcie; effect.parent = host, więc this.item = host.
  • equipToggle — respektowany: efekty modyfikacji z equipTransfer=true są pomijane gdy host jest zdjęty, chyba że wyzwalacz to właśnie equipToggle.

  overrides: { "duration.seconds": 3600 }  // trwa 1 godzinę
});

Helper szuka efektu "Trucizna" tylko dla tej modyfikacji, ignorując efekty innych modów i samej broni. Nakłada kopię na wszystkich zaznaczonych / namierzonych aktorów. Bez overrides — trwa do ręcznego usunięcia.

Efekt do czasu odłączenia moda — kompletny wzorzec apply + auto-cleanup
// ─── EFEKT 1: "Zatrucie" — trigger: manual ─────────────────────────────────
// Nałóż efekt na cel. Bez duration = trwa aż do ręcznego usunięcia LUB odłączenia moda.
await this.applyModEffectsToTargets(this.item, { name: "Zatrucie" });

// ─── EFEKT 1: "Zatrucie" — trigger: deleteEffect ───────────────────────────
// Uruchamia się gdy mod jest odłączany (propagowana kopia efektu jest kasowana z hosta).
// Usuwa wszystkie kopie "Zatrucia" nałożone przez ten konkretny efekt na aktorach.
await this.removeEffectsAppliedFromThis();

Oba skrypty są na tym samym efekcie modyfikacji — jeden z triggerem manual, drugi z deleteEffect. System automatycznie koreluje je przez UUID efektu.

Nałóż wszystkie efekty Other tej modyfikacji na cele
// Trigger: manual — nałóż wszystkie "Other" efekty tej modyfikacji
const applied = await this.applyModEffectsToTargets(this.item);
if (applied > 0) {
  this.notification(`Nałożono ${applied} efektów.`);
}
Odczyt listy efektów tej modyfikacji (UUID-based, ręczne nakładanie)
// Pobierz efekty Other tej modyfikacji, nałóż przez UUID (ścieżka natywna Foundry)
const effects = this.getModEffects(this.item, { transferType: "other" });
if (!effects.length) return;
const uuids = effects.map(e => e.uuid);
for (const target of this.getTargets()) {
  await target.applyEffect({ effectUuids: uuids });
}

Ścieżka UUID jest prostsza gdy nie potrzebujesz overrides ani automatycznego sprzątania. Uwaga: kopia na aktorze nie będzie miała flagi sourceEffect — nie zadziała wtedy removeEffectsAppliedFromThis().

Cleanup tylko na właścicielu broni (zamiast całej sceny)
// Trigger: deleteEffect — usuń tylko z aktora który posiada broń
await this.removeEffectsAppliedFromThis([this.actor]);

Nakładanie efektów przez effect.applyEffect() — styl natywny

Każdy efekt (instancja NeuroshimaActiveEffect) posiada własną metodę applyEffect(targets?, overrides?). Pozwala to nakładać efekty bezpośrednio ze skryptu, iterując po liście efektów przedmiotu — bez potrzeby używania mod-scoped helperów. Działa ze wszystkich wyzwalaczy, w tym manual.

manual Nałóż efekt przez indeks listy — styl WFRP
// Trigger: manual — "Użyj efektu"
// Nałóż pierwszy efekt z listy this.item.effects na zaznaczony cel
const effect = this.item.effects.contents[0];
if (!effect) return this.notification("Brak efektów na tym przedmiocie.", "warn");
await effect.applyEffect();  // cele: getTargetsOrSelected() automatycznie
manual Nałóż efekt po nazwie z opcjonalnym czasem trwania
// Trigger: manual — "Zatruć cel"
const effect = this.item.effects.getName("Trucizna");
if (!effect) return this.notification("Efekt Trucizna nie istnieje.", "warn");
await effect.applyEffect(null, { "duration.seconds": 3600 });  // 1 godzina
manual Nałóż efekt na konkretnego aktora (nie zaznaczenie)
// Trigger: manual — nałóż efekt na właściciela broni (nie na cel)
const effect = this.item.effects.contents[0];
if (!effect) return;
await effect.applyEffect(this.actor);
manual Iteracja listy efektów — nałóż wszystkie "Other"

const targets = this.getTargetsOrSelected();

for (const e of this.item.effects.contents) {

}
manual this.applyEffects() — skrócony zapis tej samej logiki
// Trigger: manual — ekwiwalent powyższego, zwięźlejszy zapis

await this.applyEffects(others);

this.applyEffects() automatycznie rozwiązuje cele przez getTargetsOrSelected(), wyświetla ostrzeżenie gdy brak celów i stosuje convertToApplied() dla każdego efektu (z zapisem sourceEffect).

manual Kompletny wzorzec: apply + auto-cleanup przy odłączeniu moda
// ─── Skrypt 1 na efekcie moda — trigger: manual ────────────────────────────
// Nałóż efekt "Zatrucie" na wskazany cel (z auto-cleanup gdy mod odłączony)
const effect = this.item.effects.getName("Zatrucie");
if (!effect) return this.notification("Efekt Zatrucie nie znaleziony.", "warn");
const applied = await effect.applyEffect();
if (applied) this.notification(`Zatrucie nałożone na ${applied} cel/e.`, "info");

// ─── Skrypt 2 na TYM SAMYM efekcie — trigger: deleteEffect ─────────────────
// Sprzątanie: gdy mod jest odłączany, usuń nałożone kopie efektu z aktorów
await this.removeEffectsAppliedFromThis();

Oba skrypty znajdują się na tym samym efekcie modyfikacji — różnią się tylko wyzwalaczem. Połączenie przez UUID efektu (sourceEffect) jest automatyczne.

Przykład: skrypt w weapon-mod modyfikujący atak broni-hosta

// Trigger: prepareData
// this.item = broń-host (NIE modyfikacja)
// Zwiększ atak broni o 1 gdy modyfikacja jest dołączona
if (this.item?.type !== "weapon") return;
const currentAtk = this.item.system.attack ?? 0;
await this.item.update({ "system.attack": currentAtk + 1 });
Uwaga: prepareData jest triggerem synchronicznym — nie używaj w nim await ani operacji asynchronicznych (takich jak item.update()). Używaj triggerów asynchronicznych (applyEffect, immediate, manual) do modyfikacji danych dokumentów.

Przykład: skrypt w armor-mod reagujący na obliczenie pancerza

// Trigger: armorCalculation
// this.item = pancerz-host (NIE modyfikacja)
// args.location, args.sp — bieżące obliczenia SP
